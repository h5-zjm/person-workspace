# 前后端一体化开发

## 1. 项目目标

在当前 `person-workspace` 中实现一套前后端一体化生成工作流。用户输入自然语言需求后，系统依次完成：

1. 分析需求并生成验收标准。
2. 生成结构化设计稿和可交互预览。
3. 用户确认或修改设计稿。
4. 根据已确认的设计稿生成前端页面和后端 API。
5. 自动执行静态检查、构建、接口测试和浏览器端测试。
6. 测试失败时，将错误反馈给 Codex 自动修复。
7. 所有检查通过后，输出代码、测试报告和页面截图。

第一版的目标是跑通一条稳定、可检查、可回退的完整链路，不追求多人协作、复杂工作流编排或完整 Figma 集成。

## 2. 核心原则

### 2.1 结构化设计稿是唯一事实来源

不采用“生成一张图片，再让模型根据图片猜代码”的方式。系统先生成结构化 `DesignSpec`，然后：

- 设计预览根据 `DesignSpec` 渲染。
- 前端代码根据 `DesignSpec` 生成。
- 后端接口根据 `DesignSpec` 中的实体和 API Contract 生成。
- 测试根据 `DesignSpec` 中的状态、交互和验收标准生成。
- 页面截图是验收产物，不是设计稿的唯一来源。

### 2.2 设计确认后才能生成正式代码

设计阶段只生成规格文件和预览，不修改正式业务代码。用户明确确认设计稿后，系统才允许 Codex 修改任务中已确认的目标目录。

现有 `AppServer/` 和 `BackEnd/` 是受保护目录，在任何任务中都不能被修改。`packages/ui/` 默认只读，只有用户在目标确认阶段明确授权时才能修改。

### 2.3 第一版使用单个 Codex Thread

需求分析、设计、编码、测试修复均在同一个 Codex thread 内分阶段执行，以保留上下文并降低调度复杂度。不同阶段通过 turn 级别的 `cwd` 限制工作目录，第一版不引入多 Agent 编排。

### 2.4 自动修复必须有上限

测试失败后最多自动修复 3 次。超过次数后任务进入 `FAILED`，保留错误、日志和当前代码，交由用户决定继续修复还是终止。

## 3. 项目目录和保护边界

当前 `AppServer/` 和 `BackEnd/` 不允许修改，因此工作流后端必须创建在新目录中：

| 目录 | 职责 |
| --- | --- |
| `FrontProject/` | 可修改；生成任务操作台和设计预览，也可由用户选择作为业务前端目标 |
| `apps/workflow-server/` | 新建；工作流编排、任务状态、测试执行和事件推送 |
| `apps/generator-cli/` | 新建；调用 WorkflowServer 的命令行操作入口 |
| `AppServer/` | 受保护；现有 Codex Bridge，只允许调用，不允许修改 |
| `BackEnd/` | 受保护；不允许作为生成目标，不允许修改 |
| `packages/ui/` | 默认只读；生成页面可以使用，修改前需要明确授权 |
| `generated-projects/` | 可选；存放用户确认后创建的新前端和新后端项目 |
| `.generator/` | 每次生成任务的中间产物、日志、截图和报告 |

写入规则：

- 只有用户选择“使用现有前端”时，业务页面才写入 `FrontProject/src/app/`。
- 用户选择“创建新前端”时，默认创建在 `generated-projects/<projectSlug>/frontend/`。
- 用户选择“创建新后端”时，默认创建在 `generated-projects/<projectSlug>/backend/`。
- 现有 `BackEnd/src/app/api/` 永远不作为生成目标。
- 现有 `AppServer/` 永远不承载新增 `/runs` 工作流代码。
- 优先使用 `@person-workspace/ui` 中已有组件。
- 未授权修改 `packages/ui/` 时，组件库能力不足应在目标前端内部实现业务组件，不能修改现有 UI 包。
- 不在应用层直接使用原生表单、按钮、表格等元素搭建业务界面。
- Codex 每个 turn 的 `cwd` 必须指向已确认的单个目标目录，不能使用 workspace 根目录作为编码阶段的 `cwd`。

## 5. 任务状态模型

第一版使用以下状态：

```ts
type RunStatus =
  | "PLANNING"
  | "WAITING_TARGET_CONFIRMATION"
  | "ANALYZING"
  | "DESIGNING"
  | "WAITING_DESIGN_APPROVAL"
  | "CODING"
  | "TESTING"
  | "FIXING"
  | "COMPLETED"
  | "FAILED";
```

状态转换规则：

```text
PLANNING
  -> WAITING_TARGET_CONFIRMATION
  -> ANALYZING
  -> DESIGNING
  -> WAITING_DESIGN_APPROVAL
  -> CODING
  -> TESTING
  -> COMPLETED

TESTING
  -> FIXING
  -> TESTING

任意执行阶段
  -> FAILED
```

`WAITING_TARGET_CONFIRMATION` 必须由用户确认目标目录后才能进入 `ANALYZING`。`WAITING_DESIGN_APPROVAL` 必须由用户确认设计稿后才能进入 `CODING`。

## 6. 目标目录确认

系统收到自然语言需求后，不能立即创建项目或修改代码。WorkflowServer 先判断需求需要前端、后端还是全栈，然后要求用户确认目标计划。

前端提供三个选项：

- 使用现有 `FrontProject/`。
- 创建新的前端目录。
- 当前任务不需要前端。

后端提供两个选项：

- 创建新的后端目录。
- 当前任务不需要后端。

现有 `BackEnd/` 不出现在可选项中，因为它是永久受保护目录。

默认建议：

- 只做现有页面功能时，使用 `FrontProject/`，不创建后端。
- 新的独立前端应用，创建 `generated-projects/<projectSlug>/frontend/`。
- 需要完整前后端时，同时创建 `generated-projects/<projectSlug>/frontend/` 和 `backend/`。
- 使用 `FrontProject/` 且又需要新接口时，前端目标使用 `FrontProject/`，后端目标创建在新的 `backend/` 目录。

确认数据结构：

```ts
type TargetPlan = {
  projectSlug: string;
  frontend: {
    mode: "existing" | "create" | "none";
    path: string | null;
  };
  backend: {
    mode: "create" | "none";
    path: string | null;
  };
  allowUiPackageChanges: boolean;
  protectedPaths: ["AppServer", "BackEnd"];
};
```

用户确认前，页面和 CLI 必须展示：

- 将复用哪些目录。
- 将创建哪些目录。
- Codex 可以修改哪些路径。
- 哪些目录受到保护。
- 前端和后端使用的技术栈。
- 预计执行的测试命令。

用户确认后，WorkflowServer 才创建新目录，并把 `TargetPlan` 写入任务产物。后续所有 Codex turn 和测试命令都必须以它为准。

## 7. 生成任务产物

第一版使用文件系统持久化，不引入数据库。每个任务对应一个目录：

```text
.generator/runs/<runId>/
├── run.json
├── target-plan.json
├── requirement.md
├── design-spec.json
├── api-contract.json
├── acceptance.md
├── test-report.json
├── logs/
│   ├── generation.log
│   └── test.log
└── screenshots/
    ├── design-desktop.png
    ├── design-mobile.png
    ├── implementation-desktop.png
    └── implementation-mobile.png
```

`run.json` 至少记录：

```ts
type GenerationRun = {
  id: string;
  title: string;
  requirement: string;
  status: RunStatus;
  codexThreadId: string | null;
  currentTurnId: string | null;
  retryCount: number;
  targetPlan: TargetPlan | null;
  createdAt: string;
  updatedAt: string;
  error: string | null;
};
```

## 7. DesignSpec 设计

使用 Zod 定义并校验 `DesignSpec`。Codex 设计阶段通过现有 `outputSchema` 输出符合 Schema 的 JSON。

建议的第一版结构：

```ts
type DesignSpec = {
  product: {
    name: string;
    targetUsers: string[];
    goals: string[];
  };
  pages: Array<{
    route: string;
    title: string;
    description?: string;
    layout: string;
    components: Array<{
      id: string;
      type: string;
      props: Record<string, unknown>;
    }>;
    states: Array<"loading" | "empty" | "error" | "success">;
    interactions: string[];
  }>;
  entities: Array<{
    name: string;
    fields: Array<{
      name: string;
      type: string;
      required: boolean;
      description?: string;
    }>;
  }>;
  apis: Array<{
    method: "GET" | "POST" | "PUT" | "PATCH" | "DELETE";
    path: string;
    description: string;
    request?: unknown;
    response: unknown;
  }>;
  acceptanceCriteria: string[];
};
```

设计阶段必须补齐：

- 页面路由和页面目标。
- 页面结构和共享组件选择。
- 加载、空数据、错误、成功状态。
- 关键交互和数据流。
- 数据实体和字段定义。
- 前后端 API Contract。
- 可执行的验收标准。

## 8. 设计预览

在 `FrontProject` 中实现通用 `DesignSpecRenderer`，读取 `design-spec.json` 并使用 `@person-workspace/ui` 渲染可交互预览。

第一版支持以下组件类型即可：

- PageLayout
- Card
- Form
- Input
- Textarea
- Select
- Button
- Table
- Dialog

如果设计需要的通用组件不存在：

1. 先判断是否确实属于通用能力。
2. 通用能力补充到 `packages/ui/src/components/`。
3. 仅属于当前业务的结构保留在 `FrontProject`。

设计预览至少提供：

- 桌面端和移动端视口切换。
- 页面切换。
- 加载、空数据、错误、成功状态切换。
- “修改设计”和“确认设计”操作。

## 9. AppServer 接口

在现有会话接口基础上增加生成任务接口：

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `POST` | `/runs` | 创建任务并开始需求分析 |
| `GET` | `/runs` | 获取任务列表 |
| `GET` | `/runs/:id` | 获取任务详情和当前产物 |
| `POST` | `/runs/:id/revise-design` | 根据用户意见修改设计稿 |
| `POST` | `/runs/:id/approve-design` | 确认设计稿 |
| `POST` | `/runs/:id/start-coding` | 根据已确认设计开始编码 |
| `POST` | `/runs/:id/retry` | 失败后重新执行当前阶段 |
| `POST` | `/runs/:id/interrupt` | 中断当前 Codex turn |
| `GET/WS` | `/runs/:id/events` | 推送状态、日志和 Codex 事件 |

其中 `approve-design` 和 `start-coding` 可以在后续合并；第一版先分开，便于明确设计审批和代码写入两个动作。

## 10. Codex 分阶段指令

### 10.1 需求分析阶段

输入：用户原始需求和当前仓库规则。

输出：

- `requirement.md`
- `acceptance.md`
- 尚未明确但会影响实现的问题

简单且可以安全推断的内容直接采用最少假设方案；会明显改变产品行为的问题再要求用户确认。

### 10.2 设计阶段

输入：需求、验收标准、共享 UI 组件清单和仓库规则。

输出：

- `design-spec.json`
- `api-contract.json`
- 可渲染的设计预览

此阶段禁止修改正式业务代码。

### 10.3 编码阶段

输入：已确认的 DesignSpec、API Contract、验收标准和当前代码。

输出：

- `FrontProject/src/app/` 下的业务页面。
- `BackEnd/src/app/api/` 下的业务接口。
- 必要的测试代码。
- 只有确有必要时才修改 `packages/ui`。

### 10.4 修复阶段

输入：失败命令、退出码、错误日志、失败截图和当前验收标准。

要求 Codex：

- 只修复导致验收失败的问题。
- 不顺手重构无关模块。
- 修复后重新执行完整验证。
- 同一错误连续失败时保留诊断信息，不进行无限重试。

## 11. 自动测试与验收

### 11.1 仓库基础校验

每次代码生成或修复后必须执行：

```bash
pnpm check
```

该命令覆盖：

- ESLint
- 设计规则脚本
- TypeScript
- UI 包构建
- FrontProject 构建
- BackEnd 构建
- AppServer 构建

### 11.2 后端接口测试

为生成的 API 覆盖：

- 正常请求。
- 必填参数缺失。
- 非法参数。
- 数据不存在。
- 服务异常。
- 响应状态码和 JSON 结构。

第一版优先测试 Route Handler 中可独立验证的业务逻辑，再补真实 HTTP 集成测试。

### 11.3 浏览器 E2E 测试

引入 Playwright，覆盖：

- 页面能正常打开。
- 主要表单可以提交。
- 前端能够调用生成的后端接口。
- 加载、空数据、错误和成功状态可正常展示。
- 桌面端和移动端布局无明显溢出或遮挡。
- 浏览器控制台无未处理错误。

### 11.4 视觉验收

分别对设计预览和最终实现生成桌面端、移动端截图，对比：

- 页面结构。
- 组件数量和顺序。
- 文案。
- 间距和布局。
- 颜色和设计 token。
- 关键状态。

第一版采用规则检查和模型视觉复核，不要求像素级完全一致。

## 12. 自动修复策略

```text
执行测试
  |
  +-- 全部通过 -> COMPLETED
  |
  +-- 存在失败 -> 生成失败摘要
                    |
                    v
                  FIXING
                    |
                    v
                重新执行完整测试
```

约束：

- 最大修复次数为 3。
- 每次修复都记录修改内容和测试结果。
- 只把高信号错误发送给 Codex，避免塞入完整无关日志。
- 连续出现相同错误时，第三次失败后直接进入 `FAILED`。
- 不自动执行部署、删除数据、覆盖环境配置等高风险操作。

## 13. 前端工作台

第一版页面包含以下区域：

1. 任务列表。
2. 自然语言需求输入框。
3. 当前阶段和执行进度。
4. 设计稿预览。
5. 设计修改意见输入。
6. 确认设计按钮。
7. Codex 实时事件和测试日志。
8. 最终测试报告和截图。

任务主操作保持明确：

- 需求阶段：生成设计稿。
- 设计阶段：修改设计、确认设计。
- 编码阶段：查看进度、中断。
- 失败阶段：重新执行、查看错误。
- 完成阶段：查看代码范围、测试报告和截图。

## 14. CLI 开发

需要开发 CLI，但 CLI 只是 `AppServer` 的薄客户端，不能成为第二套工作流引擎。

### 14.1 CLI 定位

```text
前端工作台 ----\
                +--> AppServer HTTP/WS --> Codex App Server
生成任务 CLI --/
```

CLI 和前端工作台调用同一套 `/runs` 接口。任务状态、Codex thread、文件持久化、测试执行和自动修复全部由 `AppServer` 负责。

CLI 的用途：

- 在前端工作台完成前验证整条生成流程。
- 创建任务并持续查看执行日志。
- 在终端中修改和确认设计稿。
- 中断、重试和检查失败任务。
- 为后续 CI、脚本化调用和批量任务提供基础入口。

CLI 不负责：

- 直接启动或连接 `codex app-server`。
- 自己维护任务状态。
- 自己保存 DesignSpec 和测试报告。
- 自己执行测试或自动修复循环。
- 绕过设计确认直接修改业务代码。

### 14.2 CLI 目录

```text
apps/generator-cli/
├── package.json
├── tsconfig.json
└── src/
    ├── index.ts
    ├── client.ts
    ├── config.ts
    ├── output.ts
    └── commands/
        ├── create.ts
        ├── list.ts
        ├── status.ts
        ├── logs.ts
        ├── revise-design.ts
        ├── approve-design.ts
        ├── retry.ts
        └── interrupt.ts
```

第一版使用 Node.js 内置 `fetch`、WebSocket 能力和 `util.parseArgs`，不额外引入 CLI 框架。只有命令数量和参数解析明显复杂后，再评估是否引入专用依赖。

### 14.3 CLI 命令

第一版需要实现：

```bash
# 创建任务
pnpm generator create --requirement "生成一个用户管理页面"

# 从 Markdown 文件读取需求
pnpm generator create --file ./requirements/user-manage.md

# 查看任务
pnpm generator list
pnpm generator status <runId>

# 查看或持续订阅日志
pnpm generator logs <runId>
pnpm generator logs <runId> --follow

# 修改和确认设计稿
pnpm generator revise-design <runId> --message "表格增加手机号字段"
pnpm generator approve-design <runId>

# 失败恢复和任务控制
pnpm generator retry <runId>
pnpm generator interrupt <runId>
```

在 workspace 根 `package.json` 增加统一入口：

```json
{
  "scripts": {
    "generator": "pnpm --filter @person-workspace/generator-cli start --"
  }
}
```

### 14.4 CLI 配置

第一版只需要以下配置：

```text
GENERATOR_SERVER_URL=http://127.0.0.1:3100
```

命令行参数 `--server` 可以临时覆盖环境变量。CLI 必须正确处理：

- AppServer 未启动。
- 请求超时。
- 返回非 JSON 内容。
- 任务不存在。
- 当前状态不允许执行操作。
- WebSocket 意外断开。

### 14.5 CLI 验收标准

- 可以通过 CLI 创建任务并获得 `runId`。
- 可以查看任务状态和已有产物。
- `logs --follow` 可以持续输出状态和 Codex 事件。
- 可以修改、确认设计稿。
- 可以中断和重试任务。
- CLI 不直接读写 `.generator/runs/`。
- CLI 不直接调用 Codex。
- 错误时返回非零退出码，并输出可理解的错误信息。

## 15. 需要开发的模块清单

最终需要开发以下六部分。

### 15.1 AppServer 生成工作流

需要实现：

- `GenerationRun` 状态模型。
- `.generator/runs/` 文件持久化。
- `/runs` HTTP 接口和事件流。
- Run 与 Codex session/thread 的关联。
- 分阶段 Codex 指令和 `outputSchema`。
- 设计审批门禁。
- 测试命令执行器。
- 最多 3 次的自动修复循环。
- 中断、恢复、重试、超时和并发写入保护。

### 15.2 FrontProject 生成任务工作台

需要实现：

- 新建任务页面。
- 历史任务列表和任务详情。
- 阶段进度和实时事件展示。

## 16. 分阶段开发计划

### 阶段一：任务模型和持久化

工作内容：

- 定义 `GenerationRun` 和状态枚举。
- 在 `.generator/runs/` 中实现文件读写。
- 增加创建、查询任务接口。
- 增加任务事件推送。
- 将任务和现有 Codex session/thread 关联。

验收标准：

- 可以创建生成任务。
- AppServer 重启后仍能读取任务记录。
- 前端可以看到任务状态变化。
- 可以从任务定位对应 Codex thread。

### 阶段二：CLI 基础操作

工作内容：

- 创建 `apps/generator-cli/`。
- 实现 AppServer HTTP 客户端。
- 实现 `create`、`list`、`status` 和 `logs --follow`。
- 增加 workspace 根 `pnpm generator` 命令。

验收标准：

- 不打开浏览器也能创建和查看任务。
- 可以持续查看任务状态和事件。
- AppServer 不可用时返回明确错误和非零退出码。

### 阶段三：需求分析和 DesignSpec

工作内容：

- 使用 Zod 定义 DesignSpec Schema。
- 通过 `outputSchema` 约束 Codex 输出。
- 生成 `requirement.md`、`acceptance.md`、`design-spec.json` 和 `api-contract.json`。
- 对无效输出进行校验并给出明确错误。

验收标准：

- 输入自然语言需求后能得到合法 DesignSpec。
- 非法 JSON 不会进入设计预览阶段。
- 设计内容包含页面、状态、交互、实体、API 和验收标准。

### 阶段四：可交互设计预览和审批

工作内容：

- 实现 `DesignSpecRenderer`。
- 支持共享 UI 的基础组件映射。
- 支持桌面端、移动端和页面状态预览。
- 增加修改设计和确认设计接口。
- 生成设计稿截图。

验收标准：

- 不写正式业务代码也能看到可交互设计稿。
- 用户修改意见能够生成新版本 DesignSpec。
- 未确认设计时不能进入编码阶段。
- CLI 和前端都可以完成设计修改与确认。

### 阶段五：前后端代码生成

工作内容：

- 编写编码阶段固定指令。
- 限制 Codex 工作目录和可修改路径。
- 根据 API Contract 先生成接口，再生成前端调用。
- 记录每次任务修改的文件列表。

验收标准：

- 页面生成在 `FrontProject/src/app/`。
- API 生成在 `BackEnd/src/app/api/`。
- 页面优先使用共享 UI。
- 前端请求和后端响应符合 API Contract。

### 阶段六：自动测试和修复闭环

工作内容：

- 接入 `pnpm check`。
- 接入后端接口测试。
- 接入 Playwright E2E。
- 生成桌面端和移动端截图。
- 实现最多 3 次的自动修复循环。
- 生成 `test-report.json`。

验收标准：

- 任意校验失败时任务不能标记为完成。
- 失败原因可以在前端查看。
- 修复后会重新执行完整测试。
- 全部检查通过后任务进入 `COMPLETED`。

### 阶段七：可靠性收口

工作内容：

- 支持中断、恢复和失败重试。
- 防止多个任务同时修改同一目标目录。
- 增加命令超时和日志大小限制。
- 增加路径校验，禁止修改工作区外文件。
- 增加任务产物清理策略。

验收标准：

- AppServer 重启后可以恢复未完成任务的信息。
- 同一目标目录同一时间只允许一个编码任务。
- 超时任务会明确失败，不会长期挂起。
- 工作区外文件不会被修改。

## 17. MVP 范围

第一版包含：

- 单用户。
- 单工作区。
- 单个 Codex thread 分阶段工作。
- 结构化 DesignSpec。
- 可交互设计预览。
- 人工设计审批。
- 前端和后端代码生成。
- `pnpm check`、API 测试、Playwright E2E。
- 最多 3 次自动修复。
- 文件系统任务持久化。
- Generator CLI 基础命令。

第一版暂不包含：

- Figma 双向同步。
- 多 Agent 并行开发。
- 数据库和任务队列。
- 多用户权限。
- 自动部署生产环境。
- 像素级视觉 Diff 平台。
- 任意技术栈和任意仓库生成。

## 18. 完成定义

一个生成任务只有同时满足以下条件才能标记为 `COMPLETED`：

- 用户已经确认设计稿。
- DesignSpec 和 API Contract 校验通过。
- 前端页面和后端 API 已写入正确目录。
- `pnpm check` 通过。
- 后端接口测试通过。
- Playwright E2E 通过。
- 桌面端和移动端截图已生成。
- 没有未处理的浏览器错误。
- 测试报告和修改文件列表已经保存。

## 19. 推荐的第一步

先完成以下四个基础能力，不直接进入代码生成：

1. `GenerationRun` 状态模型和 `.generator/runs/` 持久化。
2. Generator CLI 的 `create`、`status` 和 `logs --follow`。
3. `DesignSpec` 的 Zod Schema 和 Codex 结构化输出。
4. 使用共享 UI 实现 `DesignSpecRenderer`。

这四个能力跑通后，再接前后端代码生成和自动测试。CLI 会先成为工作流的调试入口，前端工作台和 CLI 始终复用同一套 AppServer 接口。

# 学习后端部署

## 1. 学习目标

通过把当前“股权交易中心”使用 Docker 部署到一台 Linux 服务器，理解完整上线过程：

```text
编写 Dockerfile
-> 构建镜像
-> Docker Compose 编排
-> 上传服务器并启动容器
-> 配置域名和 HTTPS
-> 查看日志
-> 更新与回滚
```

第一阶段使用：

- Linux、SSH
- Docker Engine、Docker Compose
- Dockerfile 多阶段构建
- Nginx 容器
- Java 21 JRE 容器
- Spring Boot 可执行 JAR
- uni-app H5 静态产物

暂时不引入 Kubernetes、Swarm 和复杂 CI/CD。当前项目使用单台服务器加 Docker Compose 即可。

## 3. 线上 Docker 架构

```text
用户访问 https://equity.example.com
                  |
          宿主机 80 / 443
                  |

             /             \
       / 静态页面       /api 反向代理
                              |
                  api 容器:8080

```

Docker Compose 管理两个服务：

- `web`：Nginx 和 H5，对外映射 `80/443`。
- `api`：Spring Boot，只在 Compose 内部网络暴露 `8080`。

```text
http://api:8080
```

不要把 Java 的 `8080` 映射到公网宿主机。

## 4. Docker 核心概念

### 4.1 镜像和容器

- 镜像：项目的可运行模板。
- 容器：镜像启动后的实际运行进程。
- Dockerfile：描述如何构建镜像。
- Compose：描述多个容器如何启动、联网和重启。

### 4.2 多阶段构建

构建阶段可以包含 Maven、JDK、Node.js 和 pnpm；最终运行镜像只保留 JRE 或 Nginx 以及构建产物。

### 4.3 端口与网络

- `ports`：把宿主机端口映射给容器，用于对外服务。
- `expose`：声明容器内部使用的端口。
- 同一 Compose 项目的容器可以通过服务名互相访问。

### 4.4 重启策略

```yaml
restart: unless-stopped
```

容器异常退出或 Docker 重启后会自动恢复；人工明确停止后保持停止。

## 9. Nginx 容器配置

`deploy/nginx.conf` 示例：

```nginx
server {
    listen 80;
    server_name _;

    root /usr/share/nginx/html;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location /api/ {
        proxy_pass http://api:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

这里不能写 `http://127.0.0.1:8080`。Nginx 在 `web` 容器中时，`127.0.0.1` 指向它自己，必须使用 `http://api:8080`。

## 10. compose.yaml

```yaml
name: equity-trading

services:
  api:
    build:
      context: ./JavaBackend
    restart: unless-stopped
    cpus: "1.00"
    mem_limit: 512m
    expose:
      - "8080"

  web:
    build:
      context: .
      dockerfile: FrontProject/wechat/Dockerfile
      args:
        VITE_API_BASE_URL: https://equity.example.com
    restart: unless-stopped
    cpus: "0.50"
    mem_limit: 128m
    depends_on:
      - api
    ports:
      - "80:80"
```

注意：

- `api` 不配置 `ports: 8080:8080`，避免直接暴露到公网。
- `web` 统一提供 H5 和 `/api`。
- `depends_on` 的简写形式只控制启动顺序，不代表 API 已经就绪；当前线上配置已增加 `healthcheck` 和 `condition: service_healthy`。
- `cpus` 和 `mem_limit` 是容器的使用上限，不会提前为容器预留对应资源。
- `VITE_API_BASE_URL` 是构建参数，域名变化后需要重新构建前端镜像。

## 13. 域名和 HTTPS

第一轮先用 HTTP 确认 Docker 链路，再增加 HTTPS，避免同时排查容器、Nginx、DNS 和证书。

当前已通过公网 IP 和 HTTP 完成第一轮链路验证。由于暂时没有可用域名，域名和 HTTPS 实操顺延；后续具备域名条件后，再使用 `https://域名` 重新构建前端镜像并配置证书。

## 15. 日志和排障顺序

```bash
docker compose ps
docker compose logs --tail=200 api
docker compose logs --tail=200 web
docker compose logs -f
docker compose exec web nginx -t
```

按以下顺序检查：

1. `docker compose config` 是否通过。
2. 镜像是否构建成功。
3. `api` 容器是否运行。
4. `web` 容器能否访问 `http://api:8080/api/v1/home`。
5. Nginx 配置是否正确。
6. 宿主机端口是否监听。
7. 云服务器安全组是否允许 `80/443`。
8. DNS 是否解析到正确 IP。

## 16. 更新与回滚

更新全部服务：

```bash
docker compose pull
docker compose up -d
docker compose ps
docker compose logs --tail=100
```

只更新后端：

```bash
docker compose pull api
docker compose up --no-deps -d api
```

只更新前端：

```bash
docker compose pull web
docker compose up --no-deps -d web
```

需要注意：

- `docker compose restart` 不会应用新的 Dockerfile、镜像和环境变量。
- 代码或 Dockerfile 变化后，应在本机重新构建并推送新版本镜像，再由服务器执行 `pull` 和 `up -d`。
- 失败时把 `compose.yaml` 中的镜像标签改回上一版本，再执行 `docker compose up -d <服务名>`。
- 使用不可变版本标签，例如 `equity-trading-api:0.0.2`，不要用同一标签覆盖不同内容。
- 不要只依赖 `latest` 判断当前线上版本。

## 17. 上线前必须注意的当前边界

### 17.1 当前是内存模拟数据

`JavaBackend` 的企业、项目和用户数据硬编码在 Java 服务中，没有数据库，只适合演示和学习部署。

### 17.2 当前没有真实鉴权

API 没有用户登录、Token、权限校验和操作审计。真实业务上线前必须补齐。

### 17.3 当前 CORS 只允许本地地址

`EquityController` 当前只允许 `localhost` 和 `127.0.0.1`。生产环境优先让 H5 和 API 统一通过 Nginx 同域访问。

### 17.4 不公开 API 容器端口

公网原则上只开放：

- `22`：SSH，最好限制来源 IP 并使用密钥。
- `80`：HTTP。
- `443`：HTTPS。

`api:8080` 供 Compose 内部网络访问；如需保留服务器本机排障入口，只绑定 `127.0.0.1:8080`，不要映射到公网网卡。

### 17.5 机密配置不能写入镜像

- 不要把密码、Token 或密钥写进 Dockerfile。
- 不要把机密提交到 Git。
- 使用服务器上的 `.env` 或专门的密钥管理能力。
- `.env` 必须加入 `.gitignore`。

## 21. 进阶部署阶段：负载均衡与蓝绿部署

完成单服务器 Docker Compose、域名、HTTPS、更新和普通回滚后，再进入这一阶段。

### 21.1 负载均衡

实际运行两个相同版本的 API 实例：

```text
用户请求
   |
Nginx 统一入口 80/443
   |-----------------|
api-1:8080       api-2:8080
```

需要掌握：

- 使用 Nginx `upstream` 配置多个后端实例。
- 理解轮询和最少连接等基础分配策略。
- 通过容器日志确认请求确实被分发到不同实例。
- 配合健康检查，避免继续向异常实例发送请求。
- 理解负载均衡不能代替数据库一致性、会话共享和幂等设计。

### 21.2 蓝绿部署

蓝色环境运行旧版本，绿色环境运行新版本，但用户始终访问同一个域名和 `80/443`：

```text
                     |-> blue：当前稳定版本
用户 -> Nginx 入口 --|
                     |-> green：待验证新版本
```

需要完成的实际练习：

1. 同时启动蓝色和绿色两个独立版本。
2. 先通过内部地址验证绿色版本的健康状态和关键接口。
3. 修改 Nginx 上游并平滑重新加载，把流量从蓝色切到绿色。
4. 保留蓝色版本一段时间，不立即删除。
5. 模拟绿色版本故障，把流量切回蓝色，完成零停机回滚。
6. 最后再停止和清理不再使用的旧版本。

第一轮先在单台服务器上练习。掌握真实流量切换和回滚后，再学习多服务器负载均衡、滚动发布、金丝雀发布以及是否需要 Kubernetes。
